로블록스에 번역을 다루는 방법에 대한 문서입니다.

## 소스 언어는 한국어가 기본

특별한 지시가 없는 이상, 모든 대사나 키의 소스언어는 한글이 기준입니다.
단 번역 언어로는 영어를 중심으로, 영어로도 먼저 개발을 할 수 있는 구조를 지향합니다.

해외 유저들의 기여 또한 쉽게 받기 위함입니다.

초기 개발 상태에서는 한국어로 개발하되, 안정적인 개발 궤도에 들어서면 주석과 문서 또한 영어로 바꿉니다

## 한국어가 소스 언어일지라도, 변수/출력은 영어로

로컬리제이션 테이블의 소스언어를 제외하고, 모든 변수와 출력은 영어로 작성합니다.

## 번역 스크립팅

[ExtendedTranslator](https://github.com/StoryBakery/ExtendedTranslator) 패키지를 중심으로 이용합니다

로블록스 기본 제공의 `Class.Translator` 는 한계가 많습니다.

### 로블록스 Translator 의 한계

- **번역 언어를 중간에 바꾸기 힘듭니다**

  언어가 바뀌면 새로운 `Class.LocalizationTable:GetTranslator( localeId )` 를 매번 실행해야하기 때문입니다.
  지역 변수에 할당된 개체를 바꾸는 것은 복잡한 경우가 많습니다.

- **번역언어로 존재하지 않다면 오류를 발생합니다**

  Key 에 해당하는 엔트리가 있다해도 번역 언어로 설정되어있지 않다면 소스 언어를 출력하는 것이 아닌 오류를 발생합니다.

- **로컬리제이션 테이블의 LocaleId 와 동일한 번역 텍스트를 제공해야만 함**

  위의 문제는 더욱더 심화됩니다.
  로컬리제이션 테이블의 LocaleId 와 동일한 LocaleId 로 해도,
  번역 언어로 설정되어있지 않다면 Source 를 반환하지 않으며 오류를 발생합니다.

  즉 중복된 텍스트를 적어야합니다, 이는 웹사이트에서 번역하는 경우 큰 문제가 되지 않지만,
  `csv` 파일로 번역을 하는 경우, LocaleId 와 같은 번역 언어를 지원하고 둘을 동시에 작성해둬야하는 불편함을 초래합니다.

### ExtendedTranslator 가 나은 점

그렇기에 `ExtendedTranslator` 란 래퍼 클래스를 사용하면 보다 더 확장적이고 수정이 간편하게 번역을 다룰 수 있습니다.

- 번역 언어를 중간에 바꿀 수 있음
- 기본 언어를 설정할 수 있음
- 번역 텍스트가 존재하지 않아도 오류를 발생하지 않음
- 번역 텍스트가 존재하지 않으면 그다음 하위 우선순위의 언어에서 가져옴 (e.g. 한글 -> 영어)
- 잘못된 키를 입력하면 즉각적인 에러보다 키를 반환해서 최대한 부드러운 진행을 제공함
- 객체지향으로 사용할 수 있음
- 공용 객체를 가져올 수 있음

## 로블록스에서는 Context 란 필드를 사용할 수 있지만, 보다 더 나은 가독성을 위해 사용하지 않습니다

로블록스에선 `Key` 가 아닌 `Context` 라는 인스턴스 주소를 통해 번역 타겟을 찾아내 번역하는 기능을 지원합니다.

그러나 `Context` 를 사용하면 번역 테이블의 구조가 복잡해지고,
`Key` 만을 사용하면 번역 테이블의 구조가 단순해집니다.

Bad:

```csv
Context,Source,Example,en-us
workspace.ReplicatedStorage.UI.MainGui.Greet,안녕하세요!,경쾌하게,Hello
```

Good:

```csv
Key,Source,Example,en-us
Greet,안녕하세요!,경쾌하게,Hello!
```
